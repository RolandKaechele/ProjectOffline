About

What is Project Offline?

Project Offline is a standalone desktop application designed for engineers and project leads who need a familiar project-planning environment without cloud dependencies, licensing costs, or mandatory internet connectivity.

🎯 Who is it for?

Software and systems engineers, team leads, and Project Offlines who work with existing .mpp files or need a lightweight planner that runs in air-gapped or regulated environments.

Team Leads SW Engineers Project Offlines Regulated Environments

⚙️ Technology stack

Fully Python-based with a native Qt UI. Files are read and written through MPXJ v16.1.0 (a battle-tested Java library), bridged via JPype — giving full compatibility with the MS Project file format.

Python 3.9+ PyQt5 MPXJ JPype OpenJDK 17
📂

File Compatibility

Open and save .mpp, .xml, .mpt, and .mpx project files — the same formats used by Microsoft Project.

Workflow

Getting Started

Project Offline is designed to be ready in one click. No installation wizard, no manual dependency setup.

💡
One-click launch: Double-click cli.bat (or the cli.lnk desktop shortcut). On first run it automatically downloads OpenJDK 17 and sets up the Python virtual environment — subsequent launches start in seconds.

Starting a project

1

Launch the application

Double-click the cli.lnk shortcut on your desktop or run cli.bat from the project folder.

2

Create a new project or open an existing one

New project: Use File → New (or Ctrl+N). Four things happen automatically:

  1. The project currency settings (symbol, code, decimal digits, symbol position) are read from the OS locale so the currency matches the local system from the start — e.g. € on a German Windows instead of the generic $ default.
  2. A project calendar Standard (Deutschland) is created and filled with all 9 German national public holidays for a 15-year window.
  3. An optional calendar picker lets you add regional calendars for any of the 15 German federal states or 5 international countries (France, India, Romania, China, Japan) in one step.
  4. The Project Information dialog opens — set the project start date, end date, and title before adding tasks. This prevents tasks from being incorrectly flagged as critical on an empty calendar.

Existing project: Use File → Open… (or Ctrl+O) and select any .mpp or .xml file. The Task Sheet, Gantt Chart, and Team Planner all populate automatically.

3

Switch views via the ribbon

The ribbon at the top switches between Task Sheet, Gantt Chart, Team Planner, Resource Sheet, Resource Usage, Dependencies, and Baseline. The active view button is highlighted in light blue so you always know where you are.

4

Save your changes

Use File → Save (Ctrl+S) to write back to the original file, or File → Save As… to create a new copy. Use File → Close to close the current project cleanly — you will be prompted to save any unsaved changes.

✅
Tip: The ribbon groups actions by context — the active tab determines which buttons are visible and available. The selected ribbon tab merges visually with the panel below it, and the active view button is highlighted in light blue for quick orientation. Hover over any button for a tooltip.

Resource personal calendars (populated by Confluence Calendar Sync or set manually) are reflected as semi-transparent orange blocks across the full date range of the canvas — in both daily and hourly zoom. 🗓️ Secondary (regional holiday) calendars

AD-imported resources can have a secondary holiday calendar automatically assigned based on their AD location fields (city, state, country). The matching calendar's non-working exceptions are displayed as semi-transparent yellow blocks in the Team Planner, distinct from personal vacation blocks (orange). Only exceptions unique to the secondary calendar are shown — national holidays already present in the parent calendar are filtered out. Only named exceptions in the parent calendar contribute to this filter; anonymous entries (e.g. unnamed Schulferien blocks) are excluded, so state-specific holidays that share a date with a Schulferien block are always rendered. Block labels use the exception name when present (e.g. Bayern: Heilige Drei Könige), falling back to the calendar name when the exception has no name. Hovering shows a tooltip with the exception name, date range, and source calendar name. Assignments are persisted as JSON in the project custom property "AD Secondary Calendars" and survive save/load.

Confluence Calendar Sync

Requirements: PO_09_confluence_integration.dsf

Automatically import public holidays and team vacation periods from a Confluence Team Calendar into the project's working calendars. Authentication is handled via Playwright / Chromium SSO — no passwords are stored anywhere in the application.

💡
Zero configuration after first login: The SSO session is cached in ~/.confluence_playwright_state.json (owner-readable only). Subsequent syncs run headlessly in the background — no browser window opens.
🔄
Full undo support: Before applying any change, the sync pushes a snapshot via the undo history manager. The entire sync — including stale-entry removal and expired pruning — can be reversed in a single Ctrl+Z on any view.

⚙️ Confluence Calendar Configuration dialog

Open via the RESOURCE ribbon → Confluence group, or from the Project menu. Two sections: Authentication (Manual or KeePass entry auto-fill) and Calendar Properties (Base URL, Space Key, Timezone, Days Ahead — per-project; disabled when no project is open). The Timezone field is an editable QComboBox pre-populated with all 598 IANA timezone names (via tzdata) and a contains-filter completer.

🗓️ What gets synced

Public holidays (calendar type custom or name containing holiday / feiertag) are added as non-working exceptions to the project default calendar. Only single-day events (start == end) are written; multi-day spans such as school-holiday blocks (Schulferien) are silently skipped to prevent entire weeks from being greyed out for all resources in the Team Planner.

Vacation / leave entries (calendar type leaves or name containing vacation / leave / urlaub) are matched to a named resource and added to that resource's personal calendar, creating one if needed. If a vacation event has a username that doesn't match any existing resource, a new resource is auto-created with that name — ensuring all vacation events are imported even if the project's resource list is incomplete.

Stale entries inside the sync window that no longer exist in Confluence are automatically removed. Exceptions older than 30 days are pruned from both the project and all resource calendars on every sync, keeping calendars clean over time.

The Generate… button creates a 64-byte binary key file using a cryptographically secure random source. pykeepass treats a 64-byte file as a hex-encoded 32-byte (256-bit) key, providing the same security as a full 256-bit random key.

The Create mode file dialog opens in ~/Documents. The entered path is converted to an absolute path before the database is written, so the saved QSettings path is always fully qualified — no ambiguity regardless of the working directory.

For stronger protection, leave Save password unchecked and enter the password manually on each application start.

Active Directory Integration

Requirements: PO_13_active_directory_integration.dsf

Query corporate Active Directory to look up employee details and synchronise them with the project resource sheet — no third-party LDAP library required. All queries run via PowerShell Get-ADUser, which is available on any domain-joined Windows machine with RSAT installed.

💡
Prerequisites: Windows (domain-joined), PowerShell on PATH, and the RSAT ActiveDirectory module (Add-WindowsCapability -Online -Name Rsat.ActiveDirectory.DS-LDS.Tools~~~~0.0.1.0). Call is_ad_available() first to verify — it checks both PowerShell and the module in under 1 s.

🔍 Four lookup strategies

By display name — tries exact DisplayName match first, then falls back to Surname + GivenName split on ", ".

By email (lookup_by_email_all) — fast path uses indexed mail -eq (< 1 s); wildcard mail -like '*…*' scan runs only if the exact match returns nothing.

By SAM account name (lookup_by_username_all) — fast path uses Get-ADUser -Identity (indexed, < 1 s); wildcard SamAccountName -like is skipped when the identity lookup succeeds.

Every call — whether it finds a match or not — is recorded in an unbounded lookup history list, available via get_last_lookup_results().

🔄 Resource sheet sync

sync_resources(project) iterates every resource in the project. For each resource that has a name but no email address it calls lookup_by_name() and — if found — writes the email, department, and a resource note back into the resource object. Returns a summary dict with total / updated / skipped / errors counts, consumed by the debug dump.

🛡️ Input sanitisation

All lookup functions reject strings containing PowerShell LDAP filter injection characters — parentheses, wildcard *, backslash, null byte, and shell metacharacters — and return None immediately without invoking PowerShell.

➕ Add resource from AD

The Add from AD button on the Resource Insert ribbon group lets users search AD by name, pick from a multi-result list, and create a resource in one flow. Duplicate names are hard-blocked (case-insensitive). The new resource receives an ID immediately (no save required), and a personal work calendar is created with day-of-week types copied from the default calendar — without inheriting calendar exceptions. After creation, assign_secondary_calendar_from_ad() automatically infers and persists a regional holiday calendar for the resource based on the AD user's city, state, and country fields.

Analysis

Critical Path Method (CPM)

The built-in CPM engine runs a forward+backward pass over the open project to identify the critical path — the sequence of tasks that directly determines the project finish date. Tasks with zero (or minimal) total float are highlighted red in the Gantt chart and listed in the dedicated CPM Results panel. All nine schedule dates and float values are computed entirely offline; no extra server or plug-in is required.

💡
What is the critical path? Tasks on the critical path have zero total float — any delay to one of them delays the entire project. Non-critical tasks have slack (float) that can be used without pushing the end date.

CPM Results Panel

The CPM Results tab (REPORT → Views) shows a nine-column table with Early Start / Early Finish / Late Start / Late Finish / Total Float / Free Float / Status for every task. Critical rows are highlighted red; near-critical rows are amber. Use the “Critical tasks only” checkbox to focus on blockers, or export to CSV for offline review. Double-clicking any task row opens the Task Information dialog for that task directly from the results panel.

CPM Results panel — 3 critical tasks (red), 2 near-critical (amber), 3 tasks with ample float

Open REPORT → CPM Settings to configure:

  • Display preferences — show a total-float overlay bar on Gantt bars, add a Free Float column to the Task Sheet, or toggle the CPM Results panel visibility; all settings are global and persisted in QSettings.

Integration

Jira→Project Sync

Requirements: PO_15_jira_integration.dsf

Fetch Jira issues matching a saved filter or a JQL query and automatically create or update MPXJ tasks in the current project — including a full epic hierarchy with summary/rollup container tasks for Epics. Task names are the bare Jira summary (no bracket type prefix). The entire sync is undoable with a single Ctrl+Z.

▶ run_sync() flow

run_sync(project, server, jira_client, sidecar_path) loads the existing Jira-key→UID mapping from the project sidecar, resolves the configured filter to JQL, then fetches issues in paginated pages of 100 requesting only the specific fields needed (not *all, which caused HTTP 504 timeouts on large result sets). Issues are processed in two passes: Epics first (so parent tasks exist before sub-tickets reference them), then all other issue types. Task names are set to the bare Jira summary with no type prefix.

🔄 Sidecar persistence & Jira tab

Per-task Jira metadata (jira_key, issue_type, status) is stored under the task_jira key in the project’s .custom-props.json sidecar file. On every subsequent sync _build_jira_key_lookup reconstructs the Jira-key→MPXJ Task map so issues are updated in place rather than duplicated. When a task has linked Jira data, a read-only Jira tab in the Task Information dialog shows the key, status, and a browse URL.

🏠 Epic hierarchy

Epic issues receive setSummary(True) and setRollup(True), turning them into collapsible WBS container rows in the Gantt and Task Sheet views. Sub-tickets are linked via setParentTask(epic_task). Sub-tasks whose parent epic is not in the current filter result are created at the root level without a parent.

🔒 KeePass unlock & undo support

If the configured Jira server uses auth_mode="keepass" and KeePass is locked, the user is prompted to unlock before the sync proceeds — with auto-unlock attempted first, then a master-password dialog as fallback. After a successful sync, MainWindow.run_jira_sync() pushes tasks and resources undo snapshots — a single Ctrl+Z on the Tasks or Resources view undoes the entire sync.

🔌
Configuration: Set the sync server in Settings → Jira Servers, then open Project → Jira → Configure Jira Sync to enter a JQL query or a saved filter ID. Run the sync via Project → Jira → Run Jira Sync. The filter and server selection are stored per project in the .custom-props.json sidecar under the jira2project container key.

🏖️ Vacation-aware placement

When the second segment would land inside a resource vacation block, the split algorithm shifts both seg2_start and seg_end forward past the vacation, preserving segment duration. Project non-working days are also skipped in the same pass.

🔌 Git & SVN Integration

Requirements: PO_16_version_control.dsf

Built-in version control support for both Git and Subversion (SVN) repositories. Open a project file inside a repository and a VERSION CONTROL ribbon tab appears automatically — giving you commit, history, restore, and branch operations without leaving the application.

📄 Auto-commit on save

Every Ctrl+S starts a 3-second debounce timer. After the delay, a background VcsWorker(QThread) runs the commit so the UI never blocks. Rapid saves reset the timer, preventing commit storms. A configurable message template with {project_name} and {timestamp} placeholders labels each snapshot. Scope is selectable: project file only, or all tracked changes.

🔍 Commit log & restore

VcsLogDialog shows the full commit history (newest first) with a one-click unified-diff preview panel. Select any revision and click Restore… — the module first creates a safety snapshot commit of the current state, then restores the selected revision. Replaces dated ZIP archives with proper version control.

🍀 Branch management (Git)

VcsBranchDialog lists all branches, highlights the current one, and allows creating a new branch from HEAD or switching to an existing branch. Use branches for scenario planning — an optimistic schedule on one branch, the baseline on main — then merge or discard.

🔒 Credential security

Git credentials are never passed as command-line arguments. A small helper script is generated at runtime into a tempfile.mkstemp() path and registered as GIT_ASKPASS; short-lived environment variables are deleted after the subprocess exits. SVN always passes --non-interactive; --no-auth-cache and --username/--password are added only when KeePass credentials are available, so SVN operations work without KeePass using the OS credential cache. All subprocess output is sanitised (passwords replaced with [REDACTED]) before logging. Credentials live in KeePass only — never in QSettings or sidecar JSON.

Setup: Place the project file inside a Git or SVN working copy. The VERSION CONTROL tab appears automatically on open. Open VERSION CONTROL → Settings to select the KeePass entry for remote credentials and configure auto-commit behaviour. No additional installation needed — Git and SVN binaries are bundled under tools/git/ and tools/svn/.

Guidance

Best Practices

Recommendations to get the most reliable and maintainable project schedule.

📁

Use a Work Breakdown Structure

  • Group tasks under summary tasks (work packages).
  • Keep leaf tasks short — ideally 1–5 days each.
  • Name tasks with a verb: "Implement login module", not "Login".
  • Reserve milestones (0-day tasks) for deliverable sign-offs only.
🔗

Link every task

  • Avoid floating tasks — every task should have at least one predecessor or successor.
  • Prefer Finish-to-Start (FS) unless you have a genuine overlap.
  • Use lead/lag (e.g. FS+2d) instead of manually adjusting start dates.
  • This lets the schedule recalculate automatically when dates slip.
👥

Assign all leaf tasks

  • Every leaf task should have at least one resource assigned.
  • Unassigned tasks appear in the Team Planner's chip panel — drag them to the right person.
  • Avoid assigning a resource at more than 100% capacity; monitor the Team Planner for red conflict indicators.
🏁

Save a baseline early

  • Set a baseline before the project kicks off.
  • Use the Baseline tab to track schedule variance — positive variance means slippage.
  • Re-baseline only after a formal change request is approved.
📅

Keep resource calendars current

  • Add planned vacations and public holidays to each person's resource calendar.
  • The Team Planner will show orange vacation blocks automatically.
  • The scheduler respects these when computing durations.
📊

Review the Team Planner weekly

  • Check for red resource names — they signal overloads or tasks on vacations.
  • Hover over a red name to read the detailed conflict tooltip.
  • Use drag-and-drop to resolve conflicts without leaving the view.
💾

File management

  • Save in .xml format if you need to share with non-MS-Project tools.
  • Keep a dated backup copy before major re-planning sessions.
  • Use Save As to create scenario copies without overwriting the baseline file.

⏳ Coming soon: Git integration will replace manual backups — automatic commits on save, branch-per-scenario, and full version history.

🔴

Watch the critical path

  • Focus review effort on red-highlighted critical tasks — they directly control the project end date.
  • Adding buffers (positive lag) before milestones can protect against late-arriving dependencies.
  • Adding resources to critical tasks rarely helps unless the task is effort-driven.

⏳ Modal Progress Dialogs

All long-running operations now run in a dedicated background QThread and display a modal progress dialog while they execute, keeping the main window fully responsive (repaint, resize, Ctrl+Z) throughout.

A determinate percentage bar is shown for operations that can report incremental progress (e.g. Jira → Project Sync, which reports how many issues have been processed). Non-cancellable operations (File Open, File Save) hide the Cancel button to prevent the project from being left in an inconsistent state. Cancellable operations set a cooperative cancellation flag in the worker thread and stop at the next safe checkpoint.

A timing log (last 50 entries) is maintained in memory and written to the long_running_timings key in the --debug dump, recording operation name, elapsed seconds, success flag, and ISO timestamp for performance diagnostics.
