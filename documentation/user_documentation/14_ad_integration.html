Active Directory Integration

Look up employees from your corporate Active Directory and synchronise their details — email, department — directly into the project resource sheet.

Home› Active Directory Integration

What Is Active Directory Integration?

Project Offline can query your organisation's Active Directory (AD) to look up employee details and write them back into the resource sheet without manual data entry. Four lookup strategies are supported:

  • By display name — first tries an exact DisplayName match; if not found, splits the name on ", " (Surname, Given Name) and tries a two-field match.
  • By email (lookup_by_email_all) — tries an exact mail -eq match first (indexed, typically < 1 s); only falls back to a wildcard scan if no exact match is found.
  • By SAM account name (lookup_by_username_all) — tries Get-ADUser -Identity first (indexed, < 1 s); wildcard scan is used only when the identity lookup fails.
Offline-first All AD queries use PowerShell Get-ADUser — no third-party LDAP library is required. Only the RSAT ActiveDirectory module must be installed on the workstation.

Prerequisites

RequirementDetails
Windows (domain-joined) The workstation must be joined to the same Active Directory domain you wish to query.
PowerShell on PATH PowerShell 5.1 or later. Run powershell -Command "$PSVersionTable.PSVersion" to verify.
RSAT ActiveDirectory module Install with:
Add-WindowsCapability -Online -Name Rsat.ActiveDirectory.DS-LDS.Tools~~~~0.0.1.0
(Run PowerShell as Administrator.)

Manual Lookups

Individual resources can also be looked up from the Resource Edit dialog: press Look up in AD… next to the email field to search AD by the resource's current display name. The returned email and department are pre-filled into the dialog; you can review and modify them before saving.

Tip — lookup by username If the display name lookup returns no results (e.g. the name has a localised form), edit the resource first to add the SAM account name in the Username field, then click Look up in AD… — the module will try the username strategy automatically.

Add a New Resource Directly from AD

Instead of creating a resource manually and syncing it afterwards, you can add a new resource from scratch in a single guided workflow using the Add from AD button on the Resources ribbon tab (Resource Insert group).

  1. Search — type part of the employee’s name in the search dialog. If multiple AD accounts match, a list is shown and you can select the correct person.
  2. Review — the resource dialog opens pre-filled with the AD data (display name, email, department, username). You can edit any field before confirming.
  3. Create — the resource is added to the project with a unique ID assigned immediately, so the ID column is visible right away without requiring a save/reload cycle.
  4. Calendar — a personal work calendar is created automatically. It copies the day-of-week work types from the project default calendar but does not inherit calendar exceptions (e.g. company holidays). Exceptions remain centralised on the project default calendar.
Duplicate prevention If a resource with the same name already exists in the project (comparison is case-insensitive), a warning is shown and the operation is cancelled. To update an existing resource’s AD data, use Sync from AD instead.
Automatic regional holiday calendar When a resource is imported from AD, the application reads the city, state, and country fields from the LDAP record and automatically assigns the matching regional holiday calendar (e.g. a Bavarian public-holiday calendar for resources located in Munich). The assignment is stored in the project file and is visible in the Team Planner as yellow blocks on non-working days.

Add Resources from AD Group

For bulk onboarding, use Add from AD Group from the Resources tab. The dialog lets you search AD groups, preview members, and add all selected users in one action. Duplicate resources are skipped automatically.

Security & Input Sanitisation

All inputs passed to PowerShell are sanitised before the query is built. Strings containing the following characters are rejected immediately and return None without invoking PowerShell:

Character(s)Reason
( )LDAP filter grouping — would allow filter injection.
*LDAP wildcard — would match any attribute value.
\LDAP escape prefix and PowerShell path separator.
\x00Null byte — terminates C-style strings inside LDAP libraries.

Troubleshooting

SymptomLikely cause & fix
Status bar shows AD not available Either PowerShell is not on PATH, or the RSAT ActiveDirectory module is not installed. Run
Add-WindowsCapability -Online -Name Rsat.ActiveDirectory.DS-LDS.Tools~~~~0.0.1.0
in an elevated PowerShell prompt, then restart Project Offline.
Sync hangs for a long time The domain controller may be unreachable, or a wildcard scan is running against a large directory. Check VPN / network connectivity first. If the problem persists you can raise or lower the Query timeout in Settings (default 60 s).
Resource is always skipped The resource name may not match its DisplayName in AD. Try adding the SAM account name in the Username field and re-running the sync.
Lookup returns wrong person A common name may match multiple AD accounts. Use the email or username lookup strategy by pre-filling those fields and re-syncing.
AD sync result missing from debug dump AD integration must be enabled in settings and at least one sync or lookup must have run in the current session before the dump is triggered.
