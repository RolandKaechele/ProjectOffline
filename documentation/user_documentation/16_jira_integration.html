Jira Integration

Connect to Jira Cloud or Jira Server/Data Center to synchronize project tasks with Jira issues using API Token, Password, or Personal Access Token authentication.

Home› Jira Integration

What Is Jira Integration?

Project Offline includes Jira API connectivity that allows you to configure multiple Jira servers and synchronize project tasks with Jira issues. The integration supports three authentication methods to accommodate different Jira deployment types:

  • API Token – For Jira Cloud instances (username + token via basic_auth)
  • Password – For legacy Jira servers that allow basic authentication (username + password via basic_auth)
  • Personal Access Token (PAT) – For Jira Server/Data Center with disabled basic authentication (token-only via token_auth)
Secure Credential Storage Jira credentials can be stored manually in Windows Registry (QSettings) or retrieved securely from your KeePass database. Credentials are automatically stripped of whitespace to prevent HTTP header validation errors.

Prerequisites

RequirementDetails
Jira Server Access A Jira Cloud instance or Jira Server/Data Center installation with API access enabled.
Credentials Valid credentials for the Jira server:
  • Jira Cloud: Your email address and an API Token generated from Atlassian Account Settings
  • Jira Server/Data Center with PAT: A Personal Access Token generated from your Jira profile settings
  • Legacy Jira Server: Your username and password (if basic auth is enabled)
KeePass Database (Optional) A configured and unlocked KeePass database to store credentials securely. See KeePass Credentials for setup.

Step 1 – Open Jira Sync Configuration

Open the Jira Sync Configuration dialog from:

  • TASK ribbon tab → Click Jira Sync Config button
  • Project menu → Select Jira → Jira Sync Configuration...

The dialog is divided into a server selector at the top and a tab panel below:

Server Selection

  • Select a Jira server from the dropdown. The dropdown is disabled until at least one server is configured.
  • Click Add... to open the Jira Servers management dialog. The dropdown refreshes automatically when you return.

Tab: Jira → Project

This tab controls how data is imported from Jira into your project.

JQL Filter

  • Choose JQL to enter a raw Jira Query Language string (e.g., project = MYPROJECT AND status = 'In Progress').
  • Choose Saved Filter (ID or URL) to reference a filter already saved in Jira — enter its numeric ID (e.g., 66111) or paste the full filter URL.
  • Click Test Filter to preview matching issues before saving. The test fetches up to 50 issues and shows a summary of the first 10. For saved filters, the resolved JQL is also displayed.
Project-Specific Filters The filter is stored in the project's custom properties (in the .custom-props.json sidecar file). Each project can have its own independent filter. You must have a project open to edit and save a filter — the controls are disabled otherwise.

Field Selection

The Fields to Import from Jira section shows a scrollable list of individual field checkboxes organised into 8 groups. Essential fields (issue key, summary, project key, issue type) are always imported — only the additional fields below are configurable.

GroupAvailable Fields
Core Information Project Name, Description
Status and Resolution Status, Status Progress %, Resolution, Resolution Date, Security Level
Assignment and Reporter Assignee, Assignee Display Name, Reporter, Reporter Display Name
Priority and Dates Priority, Due Date, Created Date, Updated Date
Versions and Components Components, Fix Versions, Affects Versions, Fix Version Description, Fix Version Released, Fix Version Start Date, Fix Version Release Date
Labels and Metadata Labels, Environment, Votes, Comments
Time Tracking Time Spent, Remaining Estimate, Original Estimate, Time Spent (seconds), Worklog Entries
Hierarchy and Relationships Parent Key, Epic Link, Parent Link, Sub-task Parent, Issue Links

Tick or untick individual fields as needed. The selection is saved per project in the .custom-props.json sidecar file.

First Time Setup If you haven't configured any Jira servers yet, the dropdown will be disabled. Click the Add... button to configure your first server. Once you return from the server configuration, the dropdown will update automatically.

Click OK to save. The selected server is saved to your user settings and the filter and field selections are saved to the project's custom properties for all future synchronization operations.

Step 2 – Manage Jira Servers

To configure multiple Jira servers or modify existing ones, access the Jira Servers dialog from:

  • Project menu → Select Jira → Jira Servers...
  • Jira Sync Configuration dialog → Click Add... button

In the Jira Servers dialog, you can:

  • Add a new Jira server configuration
  • Edit an existing server configuration
  • Remove a server from the list
  • Reorder servers using up/down buttons (for dropdown priority)

Step 3 – Add or Edit a Jira Server

Click Add or Edit to open the Edit Jira Server dialog. Configure the following settings:

Server Details Fields

FieldDescription
Server Name A friendly display name for the server (e.g., "EB Jira Production")
Server URL The full HTTPS URL of your Jira instance (e.g., https://jira.company.com or https://company.atlassian.net)
JQL Query A Jira Query Language (JQL) filter to limit which issues are synchronized (e.g., project = MYPROJECT AND status = 'In Progress')

Authentication Modes

Choose between Manual (enter credentials directly) or From KeePass entry (retrieve from KeePass database).

Credential Types

TypeWhen to UseRequired Fields
API Token (Jira Cloud) For Atlassian Cloud instances (e.g., *.atlassian.net) Username (email address) + API Token
Password For legacy Jira Server/Data Center with basic authentication enabled Username + Password
Personal Access Token (Server/Data Center) For Jira Server/Data Center with basic authentication disabled Token only (username not required)
HTTP 403 "Basic Authentication has been disabled"

If you receive this error, your Jira server requires a Personal Access Token (PAT). Generate a PAT from your Jira profile settings, select "Personal Access Token (Server/Data Center)" in the Credential Type dropdown, and paste the token into the credential field. Username is not required for PAT mode.

Step 4 – Test the Connection

After configuring the server details and credentials, click Test Connection to verify:

  • Server URL is reachable
  • Credentials are valid and authentication succeeds
  • The JIRA API client can retrieve user information (myself() call)

If the connection test succeeds, you will see a green success message with the authenticated username. If it fails, you will see an error message with details about what went wrong (e.g., "Invalid credentials", "Server not reachable", "Basic authentication disabled").

Debug Mode Launch the application with the --debug flag to see detailed connection test output in the console, including authentication method used (basic_auth vs token_auth), credential retrieval status, and full exception tracebacks for troubleshooting.

Using KeePass for Credential Storage

To store Jira credentials in KeePass instead of QSettings:

  1. Configure and unlock your KeePass database (see KeePass Credentials)
  2. Create an entry in KeePass with:
    • Username: Your Jira username or email (for API Token or Password modes)
    • Password: Your API Token, Password, or Personal Access Token
  3. In the Edit Jira Server dialog, select From KeePass entry
  4. Choose the Credential Type dropdown (API Token, Password, or PAT)
  5. Select your KeePass entry from the dropdown
  6. Click Test Connection to verify
Credential Stripping The application automatically strips whitespace (including trailing newlines) from usernames and credentials retrieved from KeePass to prevent HTTP header validation errors like "invalid header value b'Bearer XXXXXX\n'".

Technical Details

Authentication Methods

Credential TypeJIRA Library MethodDetails
API Token
Password		 basic_auth=(username, credential) Uses HTTP Basic Authentication with username and token/password
Personal Access Token (PAT) token_auth=credential Uses Bearer token authentication (token only, no username required)

QSettings Storage

Server configurations are stored in the Windows Registry under HKEY_CURRENT_USER\Software\ProjectOffline\ProjectManager\jira\servers as a JSON array. Each server object contains:

  • name – Display name
  • url – Server URL
  • jql_query – JQL filter (optional)
  • auth_mode – "manual" or "keepass"
  • credential_type – "token", "password", or "pat"
  • username – Username (manual mode, token/password types)
  • token – Credential (manual mode)
  • keepass_entry – Entry path (keepass mode)
  • keepass_credential_type – Credential type (keepass mode)

Module Reference

The Jira integration is implemented in src/integrations/jira_integration.py with the following public functions:

  • test_connection(server: dict) → tuple[bool, str] – Tests connectivity and authentication
  • get_jira_client(server: dict) → tuple[JIRA | None, str] – Returns authenticated JIRA client
  • get_config_summary() → dict – Returns non-sensitive config for debug dumps

Best Practices

  • Use KeePass for production credentials – Avoid storing tokens/passwords in QSettings for better security
  • Generate dedicated API tokens – Create separate tokens for each application to enable easy revocation
  • Use restrictive JQL queries – Limit synchronized issues to reduce API load and improve performance
  • Test connections after credential changes – Always verify connectivity when rotating tokens or passwords
  • Prefer PAT for Jira Server – Personal Access Tokens are more secure than passwords and work with disabled basic auth

Troubleshooting

IssueSolution
HTTP 403: Basic Authentication has been disabled Your Jira server requires a Personal Access Token (PAT). Generate a PAT from your Jira profile and select "Personal Access Token (Server/Data Center)" as the credential type.
HTTP 401: Unauthorized Invalid credentials. Verify your username and token/password are correct. For Jira Cloud, ensure you're using your email address (not username) with an API Token.
Connection timeout or "Server not reachable" Check the Server URL is correct and accessible from your network. Verify firewall settings and VPN connection if required.
Invalid header value error This is usually caused by trailing whitespace in the credential. The application automatically strips whitespace, but if the error persists, manually check your KeePass entry or manual credential field.
KeePass entry not showing up Ensure your KeePass database is unlocked and the entry exists. The dropdown only shows entries from the currently unlocked database.
