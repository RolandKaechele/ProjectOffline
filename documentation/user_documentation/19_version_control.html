Git & SVN Integration

Auto-commit on save, view commit history, restore any revision, manage branches, and sync with remote repositories — all from within Project Offline, using KeePass credentials.

Home› Git & SVN Integration

What Is Version Control Integration?

Project Offline includes built-in version control support for both Git and Subversion (SVN) repositories. When you open a project file that lives inside a repository, a dedicated VERSION CONTROL ribbon tab appears automatically. It gives you commit, history, restore, and branch operations without leaving the application.

Key benefits over manual file backups:

  • Auto-commit on save — every save captures a timestamped snapshot with zero extra effort.
  • Instant restore — return to any previous state in seconds; no hunting through dated ZIP archives.
  • Scenario branches (Git) — create a branch for a what-if schedule, keep it isolated, merge or discard later.
  • Team sync — pull changes from colleagues or push your plan to a shared remote.
Automatic Detection The VERSION CONTROL ribbon tab is hidden when no repository is detected and revealed automatically when a project file inside a Git or SVN working copy is opened. No manual activation is needed.

Prerequisites

RequirementDetails
Git or SVN binary
  • Git: bundled under tools/git/ or a system git.exe on PATH. Custom path settable in VCS settings.
  • SVN: bundled TortoiseSVN CLI binaries under tools/svn/, or a system svn.exe. Custom path settable in VCS settings.
Repository The project file must be inside a Git repository (.git/ ancestor) or SVN working copy (.svn/ ancestor). Project Offline will detect the repository root automatically.
KeePass Database (for remote operations) A configured and unlocked KeePass database with an entry for the remote repository credentials. Required only for push, pull, and SVN update; local commit and history work without credentials. See KeePass Credentials for setup.

The ribbon tab is divided into button groups. Availability of each button depends on the detected VCS type:

Step 1 – Configure VCS Settings

Click Settings in the VERSION CONTROL tab (or open it from Settings → Version Control…) to open the VCS configuration dialog.

KeePass Entry

Select the KeePass entry that holds the credentials for your remote repository (e.g. a GitHub / GitLab / Bitbucket / SVN server account). The entry is looked up by title from the open KeePass database. If KeePass is locked, a prompt to unlock it appears when a remote operation is triggered.

Auto-Commit

SettingDefaultDescription
Enable auto-commit on saveOnTrigger a commit 3 seconds after every successful project save.
Commit message templateAuto-commit: {project_name} saved at {timestamp}Placeholders {project_name} and {timestamp} are replaced at commit time.
Commit scopeProject file onlyProject file only commits just the .mpp/.xml file; All tracked changes stages all modified tracked files.

Binary Paths (Optional)

If the bundled binaries are not used, enter the full path to git.exe or svn.exe using the Browse buttons. Leave blank to use the bundled tool or the system PATH.

Committing Changes

Manual Commit

Click Commit in the ribbon. A small dialog prompts for a commit message. Click OK to commit. The result is shown in a status bar message.

Auto-Commit

When auto-commit is enabled, saving the project file (Ctrl+S or File → Save) starts a 3-second timer. After the delay, the commit runs in the background without interrupting your work. If you save again within the 3-second window the timer resets, preventing commit storms during rapid saves.

No Changes = No Commit Auto-commit is silently skipped if the repository has no staged or tracked changes since the last commit.

Viewing Commit History

Click Log / History to open the Commit Log dialog:

  • The left panel lists all commits (newest first) with hash/revision, author, date, and message.
  • Click any row to load a unified diff preview in the right panel, showing exactly what changed in that commit.
  • Select a commit and click Restore… to restore the project file to that state (see Restoring a Revision below).

Restoring a Revision

To return the project file to a previous state:

  1. Open Log / History.
  2. Select the desired commit row.
  3. Click Restore….
  4. A confirmation dialog appears. Confirm to proceed.

Before restoring, Project Offline automatically:

  • Creates a safety snapshot commit of the current state so you can always undo the restore.
  • For Git: uses git checkout <hash> -- <file> to restore only the project file, leaving other files untouched.
  • For SVN: uses svn update -r <revision> <file>.
Unsaved Changes Save the project before restoring. Any unsaved in-memory changes will be overwritten by the restore operation.

Branch Management (Git Only)

Click Branches to open the Branch Manager dialog:

  • Current branch is highlighted at the top of the list.
  • Click New Branch… to create a branch from the current HEAD and optionally switch to it immediately.
  • Select a branch and click Switch to checkout that branch. Any uncommitted changes must be committed first.
Scenario Planning with Branches Create a branch (e.g. scenario-optimistic) to explore a compressed schedule, then switch back to main for the baseline plan. Branches are isolated until you merge them.

Syncing with Remote (Pull / SVN Update)

Git — Pull (▼ Split-Button)

The Pull ribbon button is a split-button. Clicking the main area runs the default git pull (merge strategy). Click the ▼ arrow to choose a pull mode:

ModeCommandWhen to use
Pull (merge) git pull Default. Creates a merge commit if diverged from the remote.
Pull (rebase) git pull --rebase Reapplies your local commits on top of the remote. Keeps a linear history.
Fetch only git fetch --all Downloads remote changes without merging or rebasing. Useful to inspect what changed before integrating.

Credentials are supplied automatically via the GIT_ASKPASS mechanism using the KeePass entry configured in VCS Settings. All modes run in the background; progress is shown in the status bar.

Git — Commit (▼ Split-Button)

The Commit ribbon button is also a split-button. Click the ▼ arrow to access an additional mode:

ModeDescription
Commit tracked changes (default) Commits files according to the Commit scope setting in VCS Config (project file only, or all tracked changes).
Commit all tracked changes Overrides the Commit scope setting for this one operation and stages all modified tracked files. Useful as a one-off without changing the saved configuration.

SVN — Update

Click SVN Update to run svn update --non-interactive. When KeePass credentials are configured and available, --username, --password, and --no-auth-cache are also passed. When no KeePass credentials are configured, these flags are omitted so SVN can authenticate using the OS credential cache (e.g. Windows Credential Manager).

SVN — Cleanup

If an SVN update is interrupted (power loss, network failure), the working copy may become locked. Click SVN Cleanup to run svn cleanup and repair the working copy, then retry the update.

Conflict Detection & Resolution

After a Pull or SVN Update, Project Offline checks for conflicts. If conflicts are found, the Conflict Resolution dialog opens automatically and lists every conflicting file. For each file you can:

  • Resolve… — mark the conflict as resolved after you have manually edited the file.
  • Revert Selected — discard local changes and accept the incoming version.
  • Retry — re-check conflict status after external resolution.

All subprocess output is shown in the dialog for diagnosis, with credentials redacted.

Credential Security

Project Offline takes several measures to prevent credential leakage:

  • Git: Credentials are never passed as command-line arguments. A small helper script is generated at runtime into a temporary file and registered as GIT_ASKPASS. Git calls this script when it needs a username or password; the script reads the values from short-lived environment variables that are deleted after the subprocess exits.
  • SVN: --non-interactive is always passed so SVN never prompts interactively. --no-auth-cache and explicit --username/--password flags are only added when KeePass credentials are available. When no KeePass entry is configured, SVN uses the OS credential cache so that operations like svn log succeed with previously-saved credentials.
  • Log sanitisation: Before subprocess output is written to the debug log, all occurrences of the active KeePass password and username are replaced with [REDACTED].
  • KeePass-only storage: Credentials are never written to QSettings, the project file, or any sidecar JSON.

Troubleshooting

SymptomLikely CauseResolution
VERSION CONTROL tab not visible No .git/ or .svn/ directory found in the project file's ancestor chain. Ensure the project file is saved inside a Git repository or SVN working copy. Initialise a repo if needed (git init / svn checkout), then re-open the project.
Auto-commit never runs Auto-commit disabled, or repository has no changes after save. Check VCS Settings → Enable auto-commit on save. Verify the project file is tracked (git status).
Pull/Push fails with authentication error KeePass locked, wrong entry selected, or entry credentials outdated. Open KeePass, verify the entry URL and credentials, then retry. Select the correct KeePass entry in VCS Settings.
SVN update fails: "Working copy locked" A previous SVN operation was interrupted. Click SVN Cleanup in the ribbon, then retry SVN Update.
Commit blocked: "merge/rebase in progress" Git working copy is in a mid-merge or mid-rebase state. Resolve the merge/rebase manually in a terminal (git merge --abort or git rebase --abort), then return to Project Offline.
Restore did not reload the file The restore succeeded but the file was not re-opened automatically. Close and reopen the project file (File → Open Recent) to load the restored version.
