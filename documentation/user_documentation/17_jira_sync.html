Jira Sync

Two-way synchronisation between Jira and your project: import Jira issues as tasks (Jira→Project) with epic hierarchy and full undo support, and push project tasks back to Jira (Project→Jira) with dry-run preview, conflict detection, and an audit trail.

Home› Jira→Project Sync

What Is Jira→Project Sync?

The Jira→Project Sync feature runs a live synchronization between your Jira instance and the currently open project file. It reads all issues matching your configured JQL filter (or saved Jira filter) and:

  • Creates new MPXJ tasks for issues not yet in the project
  • Updates existing tasks (name, percent-complete) for issues already tracked
  • Builds an epic hierarchy — Epic issues become collapsible WBS summary rows; sub-tickets are nested beneath their parent Epic
  • Persists a mapping between Jira keys and task IDs in the project sidecar so subsequent syncs update in place rather than duplicate
Prerequisite: Jira Integration must be configured first. You need at least one Jira server configured in Jira Integration and a sync server + filter set for the current project before running a sync.

Prerequisites

RequirementDetails
Jira server configured At least one server must be added in Settings → Jira Servers. See Jira Integration.
Sync server selected A server must be selected as the sync server in the Jira Sync Configuration dialog.
Filter configured A JQL query or saved filter ID must be configured for the current project (per-project setting).
Open project file A project file must be open. The sync modifies the current in-memory project; you can save afterwards.
KeePass unlocked (if applicable) If your Jira server uses KeePass for credential storage, you will be prompted to unlock KeePass before the sync starts. You can also unlock it in advance via Settings → KeePass.

Step 1 – Configure the Sync Filter

If you have not already done so, open the Jira Sync Configuration dialog:

  • TASK ribbon tab → click Jira Sync Config
  • Project menu → Jira → Jira Sync Configuration…

In the Jira → Project tab:

  1. Select your Jira server from the dropdown (or click Add… to add one).
  2. Choose JQL and enter a JQL expression, for example:
    project = MYPROJ AND issuetype in (Epic, Story, Bug) ORDER BY created ASC
  3. Or choose Saved Filter (ID or URL) and paste the numeric filter ID or filter URL.
  4. Click Test Filter to preview matching issues and confirm the filter is correct.
  5. Click OK to save.
The filter is stored per project. Each project file can have its own independent JQL filter, stored in the .custom-props.json sidecar file alongside your project under the jira2project key.

Step 2 – Run the Sync

Once a filter is configured, run the sync:

  • TASK ribbon tab → click the main area of the Sync from Jira button (runs the normal sync), or click the ▼ drop-down arrow for advanced modes
  • Project menu → Jira → Run Jira Sync

Sync Modes (▼ Drop-Down)

The Sync from Jira ribbon button is a split-button. Click the ▼ arrow to choose a sync mode:

ModeWhen to use
Sync now (normal) Standard incremental sync — the default main-click action. Fetches all issues matching your filter and creates or updates tasks.
Changed since last sync Fetches only Jira issues that were updated after the last successful sync timestamp (force_incremental=True). Faster for large projects where most issues are unchanged.
Full resync Discards the stored Jira key → task UID mapping and re-imports all matching issues from scratch (force_full_resync=True). Use when the project structure has changed significantly and you want a clean import.
Open Jira Config Shortcut to open the Jira Sync Configuration dialog directly from the drop-down.

Sync to Jira Dry-Run

The Sync to Jira ribbon button also has a split-button drop-down. Select Dry-run preview to collect all planned Jira write operations (creates, updates, transitions) without executing any of them. A preview dialog then shows exactly what would happen, letting you review before committing.

The sync will:

  1. Connect to your Jira server using the stored credentials (prompting to unlock KeePass first if needed).
  2. Resolve the configured filter to a JQL query (if using a saved filter).
  3. Fetch all matching issues in pages of 100 (requesting only the specific fields needed to avoid timeouts on large result sets).
  4. Process Epics first, then all other issue types.
  5. Create new tasks or update existing ones — task names are set to the bare Jira summary with no type prefix.
  6. Show a summary dialog with the number of tasks created, updated, and skipped.

KeePass Unlock Prompt

If your Jira server uses KeePass for credential storage and KeePass is currently locked, a prompt appears before the sync starts:

Clicking Yes attempts auto-unlock (using a stored key file if configured). If auto-unlock fails, a master password dialog appears. Clicking No cancels the sync.

Viewing Jira Data in the Task Dialog

Once a task has been linked to a Jira issue, opening its Task Information dialog (double-click the task row) shows a Jira tab with read-only details:

FieldDescription
Jira KeyThe Jira issue key, e.g. PROJ-42
StatusThe Jira status at the time of the last sync
BrowseA clickable link that opens the issue in your browser

If no Jira data has been linked to the task yet, the tab shows an explanatory message instead.

Epic Hierarchy

When your filter returns Epic issues alongside their sub-tickets, the sync automatically builds a WBS hierarchy:

  • Epic tasks are marked as summary/rollup rows — they collapse in the Gantt Chart and Task Sheet and their duration rolls up from children.
  • Sub-tickets (Stories, Bugs, Tasks, Sub-tasks) are nested under their parent Epic using MPXJ’s setParentTask call.
  • Sub-tickets whose parent Epic is not included in the filter result are created at the root level without a parent.
Include Epics in your filter if you want the hierarchy to be built correctly. A filter that returns only Stories/Bugs without their parent Epics will create flat root-level tasks.

Percent-Complete Mapping

The sync maps Jira issue statuses to MPXJ percent-complete values:

Jira Status% Complete in Project
Done / Closed / Resolved100%
In Progress50%
To Do / Open / (any other)0%

These values are applied on every sync, so a ticket that moves from In Progress to Done in Jira will have its task updated to 100% on the next run.

Undoing a Sync

The entire sync operation is undoable with a single Ctrl+Z on the Task Sheet or Resource view. The undo stack captures a snapshot of tasks and resources immediately after the sync succeeds, so you can revert:

  • All newly created tasks
  • All updated task names, parent assignments, and percent-complete values
  • Any resource assignments that were created or modified
Save before syncing if you have unsaved changes you want to preserve independently of the sync. Undo is available only for the current session.

Sidecar File & Re-sync Behaviour

After each sync, the Jira key → task UID mapping is saved in the project’s .custom-props.json sidecar file under the task_jira key. This means:

  • Running the sync a second time will update existing tasks rather than creating duplicates.
  • If you delete a task in the project and re-run the sync, the issue will be re-created as a new task (the mapping entry was removed).
  • Issues that no longer match the filter are not deleted from the project — they remain untouched.

Debug Information

When running with the --debug flag, the debug dump (Help → Debug Dump) includes a jira_sync_result block showing the result of the most recent sync:

{
  "jira_sync_result": {
    "created": 3,
    "updated": 12,
    "skipped": 5,
    "errors": [],
    "log": [
      "[CREATE] PROJ-12 → task uid=47",
      "[UPDATE] PROJ-7  → task uid=42"
    ]
  }
}

The block is omitted when no sync has run in the current session.

Project → Jira Export Configuration

The Project → Jira tab in the Jira Sync Configuration dialog lets you configure how MS Project tasks are pushed back to a Jira server. It is complementary to the Jira → Project tab, which configures the import direction.

Full execution engine now available. Both the configuration tab and the push execution engine are fully implemented. Use the Sync to Jira button in the TASK ribbon (Jira group) to run the push with an optional dry-run preview.

Opening the Dialog

The same Jira Sync Configuration dialog is used for both directions:

  • Ribbon → TASK tab → Jira Sync Config button
  • Project menu → Jira → Jira Sync Configuration…

Once the dialog opens, click the Project → Jira tab to access export settings.

Tip: Both tabs scroll independently inside the dialog. The OK and Cancel buttons are always visible at the bottom regardless of screen size — scroll within the active tab to reach all settings.

Export Scope

Controls which tasks are included in the export run. The default is Changed since last sync.

  • Selected Tasks — only the tasks currently selected in the Gantt or Task Sheet view
  • Changed since last sync — only tasks whose data has changed since the last successful push (determined by the last_pushed_at timestamp stored in the sidecar). Default.
  • Full project — all tasks in the open project

Create / Update Mode

Controls whether the export creates new Jira issues, updates existing ones, or both. The default is Create + update.

  • Create only — create new Jira issues for project tasks that have no linked issue. Requires a project field mapping (see Field Map below).
  • Update only — update existing linked issues without creating new ones.
  • Create + update — default. Both create and update. Also requires a project field mapping.

Conflict Policy

Determines what happens when a project field value and the current Jira field value differ. The default is Prefer Jira.

  • Prefer Jira — silently keep the Jira value and skip updating that field. Default.
  • Prefer Project — overwrite the Jira value with the project value without prompting.
  • Manual review — pause on each conflict and prompt you to choose.

Dry Run

When Dry Run is checked (the default), the export is simulated without making any changes to the Jira server. The dialog shows what would be created, updated, or transitioned so you can validate your mapping before committing. Uncheck it to perform a live export.

Field Map Table

Each row in the Field Map maps a project field name (left) to a Jira field name or custom field ID (right). A checkbox on each row enables or disables that mapping for the export.

  • Use the Add button to add rows, edit cells in-place, and use Delete to remove unwanted rows.
  • Disabled rows are greyed out and their Jira field input is read-only.
  • For create-capable modes the project Jira field (lowercase) is mandatory — the dialog will warn you if it is missing or disabled.

Issue Type Map Table

Maps project task-type names (left column) to Jira issue-type names (right column, e.g. Task, Bug, Story). At least one row is required. If a Jira server is selected and reachable, the dialog performs a best-effort check to confirm that each mapped Jira issue type actually exists on that server.

Transition Map Table

Maps project task-status values (left column) to Jira transition IDs or transition names (right column). The transition map is optional; leave it empty if you do not want the export to trigger workflow transitions. When non-empty and a server is reachable, the dialog verifies that each mapped transition exists for the corresponding issue type.

Cross-Field Dependency Rules (Jira → Project Tab)

Several field checkboxes in the Jira → Project tab have a controller–dependent relationship. A dependent checkbox is automatically disabled and unchecked whenever its controller checkbox is unchecked, and re-enabled when the controller is re-checked. This prevents importing derived fields whose source data would not be available.

Examples: jira_status_percent is disabled when jira_status is unchecked; jira_assignee_display_name is disabled when jira_assignee is unchecked; all four jira_fix_version_* detail fields are disabled when jira_fix_versions is unchecked.

Saving and Validation

When you click OK, the dialog validates the Project → Jira settings before saving:

  1. At least one field-map row must be enabled.
  2. Every enabled row must have a non-empty Jira field name.
  3. Create-capable modes require an enabled project field row.
  4. At least one row must exist in the Issue Type Map.
  5. Every Transition Map row must have a non-empty transition ID or name.
  6. Best-effort server checks for issue types and transitions (skipped when the server is offline or not yet configured).

Validation errors are shown as a warning dialog; the configuration dialog stays open for you to correct them.

Per-Project Persistence

All Project → Jira settings are stored per-project inside the project2jira container of the project’s .custom-props.json sidecar file. They are completely independent of the Jira → Project settings stored in the jira2project container. The project is marked dirty and the sidecar is saved when you close the dialog with OK.

Step 3 – Push to Jira (Sync to Jira)

Once you have configured the Project → Jira tab, run the push using the dedicated ribbon button or menu item:

  • TASK ribbon tab → Jira group → click Sync to Jira
  • Project menu → Jira → Sync to Jira…
Dry-run first. The push always runs a silent dry-run pass first and shows you the Jira Push Preview dialog before writing anything to Jira. You can deselect individual rows or cancel entirely before anything is sent.

The Jira Push Preview Dialog

The preview dialog shows every action the push engine would perform:

ColumnDescription
IncludeCheckbox — uncheck to exclude a task from this push run
TaskTask name from the project
ActionCREATE, UPDATE, TRANSITION, or SKIP
Jira KeyExisting Jira issue key, or NEW for tasks that would be created
ReasonHuman-readable explanation of why the action was chosen

Actions are colour-coded: green for CREATE, blue for UPDATE, orange for TRANSITION, and grey for SKIP. Click Run Push to proceed with all checked rows, or Cancel to abort without changes.

What the Push Does

  1. Reads the Project → Jira configuration for the current project.
  2. For each included task, builds a Jira field payload from the enabled field-map rows.
  3. Unlinked tasks (no Jira key): created as new Jira issues when unlinked behavior is set to Create, or skipped.
  4. Linked tasks (have a Jira key): updated. Conflict detection compares the Jira updated timestamp with last_pushed_at — if Jira is newer and the conflict policy is Prefer Jira, the task is skipped.
  5. Status transitions: if a transition map entry matches the task’s current percent-complete bracket, the Jira workflow transition is triggered.
  6. Hierarchy: if hierarchy export is enabled, the parent task’s Jira key is set as the issue’s parent in Jira.
  7. Push metadata (task_p2j, jira_push_meta) is saved to the sidecar after a successful run.
Transient error retry. The push engine automatically retries on HTTP 429 (rate-limit), 5xx server errors, connection drops, and request timeouts using exponential back-off. Non-transient errors (e.g. 400 Bad Request, 404 Not Found) are reported immediately without retry.

Sidecar Sections for Sync to Jira

After each successful push, two sections are written to the .custom-props.json sidecar alongside the existing task_jira section:

  • task_p2j — per-task push state: Jira key (or "NEW" for unlinked tasks) and the last_pushed_at ISO timestamp.
  • jira_push_meta — run-level metadata: last_run_at, last_successful_push, and up to 20 audit_sessions records.

Audit Trail

Every completed push run appends a compact audit record to jira_push_meta.audit_sessions. Each record shows the timestamp, and the counts of created, updated, transitioned, skipped, and error items. Dry-run passes are not recorded. When the 20-entry cap is reached, the oldest entry is dropped automatically.
